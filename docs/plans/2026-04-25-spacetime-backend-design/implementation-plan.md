# Implementation Plan: Spacetime Atlas backend

> **For agentic workers:** REQUIRED SUB-SKILL. Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

_Multi-file plan. Each stage is its own file. Execute stages in order. See `patterns/multi-file-plans.md`._

**Goal:** Replace the demo-only `useTopic()` body with a real backend that turns any topic string into a `SpacetimeTopic`, served either synchronously on cache hit (<500ms) or progressively via SSE on cold-start (15s p50 / 30s p95). Every cold-start grows the graph by saving Firecrawl-fetched objects as `provisional` Objects scoped by `geom_source='spacetime_cold_start'`.

**Architecture:** Two new Django models (`SpacetimeTopicCache`, `SpacetimeQueryLog`) plus four new service modules (`spacetime_jobs`, `spacetime_resolver`, `spacetime_pipeline`, `spacetime_clusters`) under `apps/notebook/`, mounted on the existing `/api/v2/theseus/` Ninja router as a new sub-router. The cold-start pipeline mirrors the `/ask/async/` Redis pub/sub + RQ pattern verbatim; the channel name is `theseus:spacetime:<job_id>`. Bucket scoring calls the existing `theseus-spacetime-infer` Modal app (H100). LLM chrome calls `SPEAKING_26B_URL` via `call_speaking_service('26b', ...)`. Firecrawl ingestion reuses `provisional_ingest.ingest_provisional_page` with the spacetime-specific `geom_source` value.

**Tech Stack:** Django 5.x, Django Ninja, RQ, Redis pub/sub, pgvector (`VectorField` on `SpacetimeTopicCache.title_embedding`), `ArrayField` for `object_ids`, Modal H100 (`theseus-spacetime-infer`), 26B GGUF via `SPEAKING_26B_URL`, Tavily/Firecrawl via `web_fetch.fetch_page_content` and `provisional_ingest.ingest_provisional_page`. Frontend: Next.js client component using `EventSource` for SSE.

## Overview

When a user navigates to `/spacetime?q=carthaginian-salt-trade`, the frontend posts `{query: "carthaginian-salt-trade"}` to `POST /api/v2/theseus/spacetime/topic/`. The view runs the deterministic 5-step resolver ladder (slug, title, substring, SBERT, fallback). On cache hit it returns the full `SpacetimeTopic` JSON. On miss it enqueues `spacetime_cold_start_task` to RQ and returns `{job_id, stream_url, status_url}`. The worker executes seven bounded stages, publishing `stage`, `cluster`, `chrome`, `complete`, and `error` events to `theseus:spacetime:<job_id>`. The frontend subscribes via `EventSource` and renders dots progressively.

Cluster notes are extracted verbatim from `Claim` rows that link to objects in each bucket; only the topic chrome (`title`, `sub`, `era_band`, `mode`) is generated by the 26B. Provisional objects from Firecrawl carry `geom_source='spacetime_cold_start'` so a one-line filter recovers a bad batch.

## Execution order

Stages run sequentially in the order listed above. The executor re-reads each stage's file JUST BEFORE dispatching its tasks, since earlier stages may have surfaced discoveries that amend downstream stages (JIT discipline). Stage 5 (`ml-pro`) is the only stage that touches Modal-side concerns; everything else is Django-side and stays inside `django-engine-pro`. Stage 7 is the only stage that touches the Website repo.

There is no `cross-cutting.md` for this plan: every concern fits cleanly inside a single stage.
