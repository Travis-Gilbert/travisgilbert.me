# Connection Feedback UI: Training the Engine Through Natural Use

## The Problem

The Level 2 learned scorer (`learned_scorer.py`) is fully built. It expects
`ConnectionFeedback` records with six labels:

| Label | Signal Type | Source |
|---|---|---|
| `engaged` | Implicit positive | User interacted meaningfully with a connection |
| `manual` | Explicit positive | User created a connection themselves |
| `web_validated` | Automated positive | Background web search confirmed co-occurrence |
| `dismissed` | Explicit negative | User rejected a connection |
| `ignored` | Implicit negative | Connection was shown but user did not interact |
| `web_unvalidated` | Automated negative | Background web search found no co-occurrence |

The `web_validated` and `web_unvalidated` labels are generated by a background
RQ task (no UI needed). The `manual` label is generated when a user creates a
connection through `ConnectionComposer` (already built, needs a signal hook).

That leaves three labels that require UI-level changes: `engaged`, `dismissed`,
and `ignored`. Of these, `engaged` and `ignored` should be captured through
natural behavior (implicit), and `dismissed` should be a lightweight explicit
action.

**Design principle**: The user should never feel like they are rating
connections. They should feel like they are using CommonPlace, and the engine
should learn from how they use it.

## Current Connection Surfaces

Connections appear in five places in the UI. Each one is a feedback opportunity.

| Surface | Component | What User Sees | Current Interaction |
|---|---|---|---|
| Object detail | `ConnectionList.tsx` | Edges with direction, type, reason, strength | Click navigates to target object |
| Compose dock | `ComposeDiscoveryDock.tsx` | Live discoveries as you write | Click opens discovered object |
| Knowledge map | `KnowledgeMap.tsx` | D3 force graph with edge lines | Click node to select, drag to rearrange |
| Resurface | `ResurfaceView.tsx` | Forgotten objects with explanation | Click to revisit |
| Grid/Timeline | `ObjectRenderer` (module variant) | Edge count badge | No interaction with edges |

## Feedback Design: Three Layers

### Layer 1: Implicit Engagement (generates `engaged` labels)

These signals are captured automatically when the user interacts with
connections in ways that indicate the connection was useful.

**1a. Dwell on connection detail**

When a user hovers or focuses on a connection card in `ConnectionList` for
more than 1.5 seconds, that is engagement. They read the reason. They
considered the connection. This is a weak positive signal.

Implementation: Add an `IntersectionObserver` + timer to each connection card.
When the card has been visible in the viewport AND hovered for 1.5s, fire a
feedback event.

```typescript
// Feedback event shape (sent to API)
interface ConnectionFeedbackEvent {
  from_object_id: number;
  to_object_id: number;
  edge_id: number;
  label: 'engaged' | 'dismissed';
  label_strength: number;  // 0.0 to 1.0
  surface: 'connection_list' | 'compose_dock' | 'knowledge_map' | 'resurface';
  interaction: string;     // 'dwell' | 'navigate' | 'dismiss' | 'drag_pin' | 'manual_create'
}
```

Label strength for dwell: `0.3` (weak signal).

**1b. Navigate through a connection**

When a user clicks a connection in `ConnectionList` and actually navigates to
the target object, that is a stronger positive signal. They found the
connection interesting enough to follow.

This already happens (click handler in ConnectionList calls `onOpenObject`).
The change: fire a feedback event alongside the navigation.

Label strength for navigate: `0.6` (moderate signal).

**1c. Navigate through a compose discovery**

When a user clicks a discovered object in `ComposeDiscoveryDock`, they are
saying "this discovery is relevant to what I am writing." This is a strong
positive signal because it happened during active composition.

Label strength for compose navigation: `0.8` (strong signal).

**1d. Pin a discovered connection**

When a user drags a discovery result from the compose dock and pins it to
their object (using the existing Lego composition system), that is the
strongest implicit positive signal. They are incorporating the engine's
discovery into their own work.

Label strength for pin: `0.9` (very strong signal).

**1e. Follow an edge in the knowledge map**

When a user clicks a node in `KnowledgeMap` that they reached by visually
following an edge from a previously selected node, that is engagement with
the edge between them.

Implementation: Track the previously selected node. When a new node is
selected and an edge exists between the previous and new selection, fire
a feedback event for that edge.

Label strength for map navigation: `0.5` (moderate signal).

### Layer 2: Explicit Dismissal (generates `dismissed` labels)

A single, lightweight action that says "this connection is noise."

**2a. Swipe-to-dismiss on connection cards**

In `ConnectionList`, add a subtle swipe-left gesture (or a small X button
that appears on hover) that dismisses a connection. The card animates out.
The edge is soft-hidden from the user's view (not deleted from the graph).

The dismiss action creates a `ConnectionFeedback` record with label
`dismissed` and strength `0.8`.

Visual design: On hover, a small muted X appears at the right edge of the
connection card, in the same position as the strength dots. Click it, the
card fades and slides right. A toast appears: "Connection hidden. Undo?"
with a 5-second undo window.

```
+----------------------------------------------+
| <- PersonPill: Dario Amodei                  |
|    semantic_similarity                       |
|    "Both discuss AI safety governance"  [x]  |  <- dismiss button
|    ...                                       |
+----------------------------------------------+
```

**2b. Dismiss in compose dock**

In `ComposeDiscoveryDock`, add the same dismiss gesture. When a live
discovery result is not relevant, the user can dismiss it. The dismissed
result fades out and does not reappear for the current compose session.

This is especially valuable because compose discoveries happen in real-time
during active writing. The user has strong context for whether a discovery
is relevant or noise.

**2c. Dismiss edge in knowledge map**

When an edge is selected in the knowledge map (by clicking it or by
selecting both connected nodes), show a small floating action: "Hide this
connection." The edge fades from the visualization and a feedback record
is created.

### Layer 3: Passive Ignore (generates `ignored` labels)

These signals are captured when connections are shown but the user does not
interact with them at all.

**3a. Impression tracking in ConnectionList**

When `ConnectionList` renders and the user scrolls past connections without
clicking, hovering for >1.5s, or dismissing, those connections are recorded
as `ignored` after the ObjectDrawer closes or after a 30-second timeout.

Implementation: When ConnectionList mounts, record which edge IDs were
displayed (the "impression set"). When the drawer closes or 30 seconds
elapse, compare the impression set against the engaged/dismissed sets.
Any edge that was shown but not interacted with gets an `ignored` label.

Label strength for ignore: `0.2` (weak negative signal). The weakness
accounts for the fact that the user might not have actually seen the
connection (it may have been below the fold).

**3b. Impression tracking in compose dock**

Same pattern for `ComposeDiscoveryDock`. Discoveries that appeared in the
dock for the duration of a compose session but were never clicked or
dismissed are recorded as `ignored`.

Label strength: `0.3` (slightly stronger than ConnectionList because compose
discoveries are more prominently displayed and contextually relevant).

## API Endpoint

### POST /api/v1/notebook/feedback/

```json
{
  "from_object_id": 42,
  "to_object_id": 87,
  "edge_id": 156,
  "label": "engaged",
  "label_strength": 0.6,
  "surface": "connection_list",
  "interaction": "navigate"
}
```

Response: `201 Created` with the ConnectionFeedback record.

The endpoint must:
1. Find or create the ConnectionFeedback record (unique on from_object + to_object + label)
2. Capture the feature vector AT EVENT TIME by calling `build_feature_vector(from_obj, to_obj)`
3. Store the surface and interaction metadata for later analysis
4. If a record with the same (from_object, to_object, label) already exists, update the
   `label_strength` to the maximum of old and new values (engagement compounds, it does not reset)

### POST /api/v1/notebook/feedback/batch/

For impression tracking (ignored labels), send in bulk when the drawer closes:

```json
{
  "events": [
    { "from_object_id": 42, "to_object_id": 87, "edge_id": 156, "label": "ignored", "label_strength": 0.2, "surface": "connection_list", "interaction": "impression_timeout" },
    { "from_object_id": 42, "to_object_id": 23, "edge_id": 201, "label": "ignored", "label_strength": 0.2, "surface": "connection_list", "interaction": "impression_timeout" }
  ]
}
```

## Component Changes

### ConnectionList.tsx

Current: click handler navigates to target object.

Add:
- `onConnectionEngaged` callback prop (called on navigate, with edge data)
- `onConnectionDismissed` callback prop (called on dismiss)
- Hover timer (1.5s dwell = weak engaged signal)
- Dismiss button (appears on hover, right-aligned)
- Impression tracking (record displayed edge IDs, fire ignored on unmount)
- Toast for dismiss with undo

New props:

```typescript
  onConnectionFeedback?: (event: ConnectionFeedbackEvent) => void;  // NEW
}
```

### ComposeDiscoveryDock.tsx

Current: click handler opens discovered object.

Add:
- Dismiss button per discovery module (small X, appears on hover)
- Navigate fires engaged feedback with surface='compose_dock'
- Session-level impression tracking for ignored labels
- Dismissed results fade out and are excluded from re-rendering

### KnowledgeMap.tsx

Current: click selects node, no edge interaction.

Add:
- Track previously selected node
- On new selection, check if an edge exists between prev and current
- If yes, fire engaged feedback for that edge
- Edge click/selection shows a small floating dismiss action
- Edges with dismissed feedback render with reduced opacity (0.2)

### ObjectDrawer.tsx

The drawer is the container for ConnectionList. It manages the lifecycle.

Add:
- Pass `sourceObjectId` to ConnectionList
- Provide `onConnectionFeedback` handler that calls the API
- On drawer close: fire batch ignored events for unengaged impressions
- Debounce feedback calls (100ms) to avoid spamming during rapid browsing

## Visual Design

### Dismiss button on connection cards

The dismiss button should feel like a natural part of the card, not an
intrusive rating widget. It appears on hover in the strength dots position:

```
Default state (no hover):
+---------------------------------------------------+
| -> Dario Amodei                                   |
|    semantic_similarity  "Both discuss AI safety"   |
|                                              ...   |  <- strength dots
+---------------------------------------------------+

Hover state:
+---------------------------------------------------+
| -> Dario Amodei                                   |
|    semantic_similarity  "Both discuss AI safety"   |
|                                               x   |  <- dismiss replaces dots
+---------------------------------------------------+

Dismissed (animates out):
+---------------------------------------------------+
|    (slides right, fades, then removed from list)   |
+---------------------------------------------------+
```

CSS: The dismiss button uses `var(--cp-text-faint)` at rest, `var(--cp-red)`
on hover. Size: 14x14px. The card uses `transition: transform 200ms, opacity 200ms`
for the dismiss animation.

### Dismiss toast

```
+------------------------------------------+
| Connection hidden.  [Undo]               |
+------------------------------------------+
```

Uses the existing Sonner toast system. "Undo" reverses the dismiss and
deletes the feedback record via `DELETE /api/v1/notebook/feedback/{id}/`.

### Compose dock dismiss

In the discovery dock, dismissed items fade out with `opacity: 0, height: 0`
over 200ms. The dock re-flows. No toast needed (compose is fast-paced,
toasts would be disruptive).

### Knowledge map edge feedback

When an edge is engaged (user navigated along it), the edge briefly pulses
with the terracotta accent color (`var(--cp-red)`, 500ms pulse, then fades
back). This provides visual feedback that the engine noticed the interaction.

When an edge is dismissed, it fades to 20% opacity over 300ms. A subtle
strikethrough pattern appears on the edge line.

## Signal Strength Summary

| Interaction | Label | Strength | Surface |
|---|---|---|---|
| User creates connection manually | `manual` | 1.0 | ConnectionComposer |
| User pins a compose discovery | `engaged` | 0.9 | ComposeDiscoveryDock |
| User clicks compose discovery | `engaged` | 0.8 | ComposeDiscoveryDock |
| User explicitly dismisses | `dismissed` | 0.8 | Any surface |
| User navigates via connection | `engaged` | 0.6 | ConnectionList |
| User follows edge in map | `engaged` | 0.5 | KnowledgeMap |
| User dwells 1.5s on connection | `engaged` | 0.3 | ConnectionList |
| Connection shown, not interacted | `ignored` | 0.2-0.3 | Any surface |

## Data Volume Estimates

With typical use (10-20 objects viewed per session, 3-5 connections per object):

- ~50 impression events per session (mostly `ignored`)
- ~5-10 engagement events per session
- ~1-2 dismiss events per session
- ~0-1 manual create events per session

At daily use, this generates ~50-70 feedback records per day. The scorer
needs 50 records to begin training (reachable in 1-2 days) and 200 records
for full learned scoring (reachable in 3-5 days).

The batch endpoint for ignored labels keeps the API call count low (one
call per drawer close rather than one per impression).

## Invariants

1. **Feature vectors are captured at event time.** Never reconstructed later.
   Model parameters may change, but the training data reflects the state of
   the engine at the moment the user interacted.

2. **Feedback is additive, never destructive.** Dismissing a connection hides
   it from the user's view but does not delete the Edge from the graph. The
   Edge remains available for the scorer to learn from.

3. **Implicit signals outnumber explicit signals.** This is by design. The
   user should generate 5-10x more implicit feedback than explicit feedback.
   The `label_strength` field compensates for the noisiness of implicit signals.

4. **No feedback fatigue.** The user never sees a prompt asking them to rate
   a connection. All explicit feedback (dismiss) is initiated by the user,
   not requested by the system.

5. **Undo for all destructive actions.** Dismiss always offers a 5-second
   undo window. This is both good UX and good data quality (accidental
   dismissals can be reversed before they corrupt the training data).
